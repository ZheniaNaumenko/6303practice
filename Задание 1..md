# анализ конкурентов

### Level.Travel

![[Pasted image 20260619124941.png]]

**Анализ Level.Travel с помощью DevTools:**

- **Сильные стороны (Backend):** Поисковый API работает очень быстро. Время получения данных о турах составляет **96–205 мс** (0.1–0.2 секунды). Это говорит о хорошо оптимизированной серверной части и быстром отклике на запросы пользователя.
- **Слабые стороны (Frontend):** Общее время загрузки страницы составило **55 секунд** при объеме переданных данных **22 МБ** и **429 запросах**.
- **Выявленная проблема:** Несмотря на быстрый поиск, интерфейс перегружен тяжелыми ресурсами (вероятно, изображениями отелей высокого разрешения, которые загружаются без должной оптимизации или lazy-loading). Это увеличивает время первого входа и расходует трафик пользователя.

### OnlineTours.ru

![[Pasted image 20260619125620.png]]
**Анализ Onlinetours  с помощью DevTools:**

- **Сильные стороны (Frontend):** Легкая страница с минимальным количеством запросов. Общее время загрузки составило **19 секунд** при объеме данных всего **3-6 МБ** и **93 запросах**. Это в разы меньше, чем у конкурентов.
- **Слабые стороны (Backend):** Поисковый API работает медленно. Время выполнения запросов `get_result` составляет **0.9-1.1 секунды** (904 мс – 1.14 с), что в **5-10 раз медленнее**, чем у Level.Travel.
- **Выявленная проблема:** Серверная часть не оптимизирована для быстрой обработки поисковых запросов. Пользователь долго ждет результаты поиска, что увеличивает время подбора тура и может привести к отказу от бронирования. Также наблюдается большое количество запросов к аналитике (`collect?`) и трекерам, что создает лишнюю нагрузку.
### Travelata

![[Pasted image 20260619125713.png]]
**Анализ Travelata с помощью DevTools:**

- **Сильные стороны:** Сбалансированные показатели. Поисковые запросы выполняются за **~1 секунду**, что является средним показателем между конкурентами.
- **Слабые стороны:**
    - Долгая полная загрузка страницы — **39 секунд**
    - Большой объем ресурсов — **23 МБ**
    - Много повторяющихся запросов — **356 запросов**, включая множественные одинаковые запросы к `checkbox` (видны на скриншоте)
- **Выявленная проблема:**
    1. **Неэффективная работа с ресурсами** — система загружает множество одинаковых элементов (checkbox), что указывает на отсутствие кэширования и дублирование кода
    2. **Перегруженность трекингом и аналитикой** — много запросов к `collect?`, `envelope?`, `profile`, что замедляет работу
    3. **Отсутствие оптимизации фронтенда** — 23 МБ данных при 356 запросах создают избыточную нагрузку на браузер
### ViPN Travel

![[Pasted image 20260619164550.png]]

**Анализ ViPN Travel с помощью DevTools:**

- **Сильные стороны (Backend):** Поисковый API работает относительно быстро. Запросы `modresult.php` (основной поиск) выполняются за **67-191 мс**, что сопоставимо с Level.Travel. Общий объем переданных данных небольшой — **5.8 MB**.
- **Слабые стороны (Frontend):** Несмотря на быстрый API и небольшой объем данных, общая загрузка страницы составила **18 секунд**. Количество запросов — **126** (средний показатель).
- **Выявленная проблема:** Диспропорция между быстрым бэкендом и медленной общей загрузкой указывает на проблемы с оптимизацией рендеринга и загрузки ресурсов (CSS, JS, изображения). Вероятно, отсутствует асинхронная загрузка скриптов и ленивая загрузка изображений.
### Tours.Tutu.ru

![[Pasted image 20260619164630.png]]

**Анализ tours.tutu с помощью DevTools:**
- **Сильные стороны (Frontend):** Умеренное время полной загрузки страницы — **21 секунда** при достаточно большом объеме данных **26 MB**.
- **Слабые стороны (Backend):**
    1. Поисковый API работает нестабильно — запрос `tours?limit=20000...` завершился с **ошибкой 500** и временем **3.06 секунды**
    2. Запросы `searchMinPriceByCriteria` выполняются за **85-201 мс**, что приемлемо, но есть дублирование запросов (5 одинаковых запросов)
    3. Большое количество запросов к аналитике (`collect?` — 515 мс, 506 мс)
- **Выявленная проблема:**
    1. **Нестабильность API** — ошибки 500 критичны для пользовательского опыта
    2. **Неэффективные запросы** — дублирование `searchMinPriceByCriteria` и избыточный лимит `limit=20000` создают ненужную нагрузку
    3. **Перегруженность трекерами** — много запросов к метрике и аналитике замедляют работу

Использовался популярный запрос: Турция, 7 ночей, 2 взрослых

## Выводы

Проведенный анализ пяти ведущих сервисов поиска и бронирования туров (Level.Travel, OnlineTours.ru, Travelata, ViPN Travel, Tours.Tutu.ru) выявил **критические проблемы**, снижающие качество пользовательского опыта и конверсию бронирования:

**1. Дисбаланс производительности:** ни один сервис не обеспечивает одновременно быстрый бэкенд и оптимизированный фронтенд. Level.Travel демонстрирует отличный API (0.1–0.2 сек), но загружает 22 MB данных за 55 секунд. OnlineTours, напротив, имеет легкую страницу (3–6 MB), но медленный поиск (0.9–1.1 сек).

**2. Отсутствие интеллектуальных функций:** 3 из 5 сервисов не имеют системы рекомендаций, 3 из 5 — персонализации, **ни один сервис не использует историю просмотров** для улучшения поиска. Это приводит к тому, что пользователи вынуждены вручную применять 15–20 фильтров и просматривать сотни нерелевантных вариантов.

**3. Неэффективная работа с данными:** дублирование запросов (Travelata загружает одинаковые checkbox многократно), избыточные лимиты (Tours.Tutu запрашивает `limit=20000`), перегруженность трекерами аналитики создают излишнюю нагрузку и замедляют работу.

**4. Ограниченная функциональность фильтров:** даже при наличии 15–20 фильтров, они работают медленно (11–25 сек) и не используют машинное обучение для предсказания предпочтений пользователя.

На основе выявленных проблем **формулируется интеллектуальная бизнес-функция**: **«Адаптивная система персонализированного подбора туров с предиктивной загрузкой контента на основе машинного обучения»**, которая решает две ключевые задачи: (1) внедряет ML-алгоритмы для анализа поведения пользователя, истории просмотров и предпочтений с целью формирования персонализированной выдачи туров, что устраняет необходимость ручного применения множества фильтров; (2) реализует предиктивное кэширование и адаптивную загрузку ресурсов, подгружая только те карточки туров и изображения, которые с высокой вероятностью заинтересуют конкретного пользователя, что сократит объем передаваемых данных с 22 MB до 2–3 MB и ускорит работу интерфейса в 5–10 раз. Данная функция обеспечит конкурентное преимущество за счет сочетания высокой производительности и интеллектуального подхода, которого нет у существующих игроков рынка.

# Глоссарий

**Турпакет** — комплекс туристических услуг (перевозка, размещение, питание, трансфер), продаваемый по единой цене через туроператора.

**Туроператор** — компания, формирующая туристические продукты и реализующая их через сеть турагентов или напрямую клиентам.

**Динамическое пакетирование** — технология автоматического формирования тура из отдельных компонентов (перелёт + отель + трансфер) в реальном времени на основе актуальных предложений.

**Конверсия** — ключевая бизнес-метрика, отражающая отношение количества завершённых бронирований к общему числу поисковых запросов.

**API (Application Programming Interface)** — программный интерфейс для взаимодействия между клиентской частью сервиса и сервером туроператора. Через API сервисы получают актуальные данные о турах.

**Backend (Серверная часть)** — серверная архитектура системы, отвечающая за обработку бизнес-логики и взаимодействие с внешними API.

**Frontend (Клиентская часть)** — пользовательский интерфейс веб-приложения.

**Кэширование** — механизм сохранения результатов предыдущих запросов для их повторного использования без обращения к серверу.

**Lazy Loading (Ленивая загрузка)** — техника оптимизации, при которой ресурсы загружаются только при попадании в область видимости пользователя.

**Машинное обучение (Machine Learning, ML)** — раздел искусственного интеллекта, изучающий методы построения алгоритмов, способных обучаться на данных. 

**Рекомендательная система** — информационная система, фильтрующая данные на основе предпочтений пользователя и предлагающая релевантные туры. В конкурентном анализе присутствует только у ViPN Travel и Tours.Tutu.

**Персонализация** — адаптация поисковой выдачи под конкретного пользователя на основе его истории просмотров и поведения. Реализована только у Travelata и Tours.Tutu.

**Collaborative Filtering (Коллаборативная фильтрация)** — метод рекомендаций, основанный на анализе поведения схожих пользователей.

**Content-Based Filtering (Контентная фильтрация)** — метод рекомендаций, при котором система анализирует характеристики объектов (звёздность отеля, тип питания) и сопоставляет их с профилем предпочтений пользователя.

**Предиктивная аналитика** — использование статистических моделей и ML-алгоритмов для прогнозирования будущих событий (изменение цен, вероятность бронирования).

**Бизнес-функция** — логически завершённая единица функциональности продукта, решающая конкретную бизнес-задачу и приносящая измеримую ценность.

**User Story (Пользовательская история)** — краткое описание функциональности в формате: «Как [роль], я хочу [действие], чтобы [ценность]».

**Definition of Ready (DoR)** — чек-лист критериев, которым должна соответствовать задача перед началом работы.

**Definition of Done (DoD)** — чек-лист критериев, выполнение которых означает полную завершённость задачи.

**Бизнес-цель** — измеримый результат, которого стремится достичь компания (например, «сократить время подбора тура с 30 до 5 минут»).

**Конкурентный анализ** — систематическое исследование продуктов конкурентов для выявления сильных и слабых сторон и формирования конкурентных преимуществ.


# бизнес-цели

Основной бизнес-целью является разработка интеллектуальной бизнес-функции персонализированного подбора и бронирования тура на основе машинного обучения, которая устраняет выявленные в конкурентном анализе недостатки существующих сервисов (Travelata, Level.Travel, OnlineTours, ViPN Travel, Tours.Tutu) в части производительности, персонализации и качества пользовательского опыта.

### Конкретные бизнес-цели (SMART-подход)

### Цель 1. Сокращение времени подбора тура

- **Показатель:** Среднее время от начала поиска до выбора тура
- **Текущее состояние (по данным конкурентного анализа):**
    - Level.Travel: 48 сек (полная загрузка) + время на просмотр нерелевантных результатов
    - Travelata: 45 сек
    - Tours.Tutu: 22 сек
    - OnlineTours: 16 сек
    - ViPN Travel: 15 сек
- **Целевое значение:** ≤ 5 минут от начала поиска до бронирования
- **Обоснование:** Пользователи вынуждены вручную применять 15-20 фильтров и просматривать сотни вариантов (от 100 до 914 отелей в выдаче)

### Цель 2. Оптимизация производительности фронтенда

- **Показатель:** Объём передаваемых данных и время загрузки страницы
- **Текущее состояние (по данным конкурентного анализа):**
    - Tours.Tutu: 26 MB, 22 сек
    - Travelata: 23 MB, 45 сек
    - Level.Travel: 22 MB, 55 сек
    - ViPN Travel: 5.8 MB, 18 сек
    - OnlineTours: 3-6 MB, 19 сек
- **Целевое значение:** ≤ 3 MB данных, ≤ 5 секунд полная загрузка
- **Обоснование:** Предиктивная загрузка только релевантных туров сократит объём данных в 7-8 раз за счёт ML-моделей и lazy-loading

### Цель 3. Снижение времени ответа поискового API

- **Показатель:** Время выполнения поискового запроса
- **Текущее состояние (по данным конкурентного анализа):**
    - Level.Travel: 96-205 мс
    - ViPN Travel: 67-191 мс 
    - Tours.Tutu: 85-201 мс, но с ошибками 500
    - Travelata: ~1000 мс 
    - OnlineTours: 904-1140 мс 
- **Целевое значение:** ≤ 200 мс при 100% стабильности (без ошибок 500)
- **Обоснование:** Кэширование популярных запросов и предиктивная аналитика

### Цель 4. Внедрение полноценной персонализации

- **Показатель:** Наличие и глубина персонализированной выдачи
- **Текущее состояние (по данным конкурентного анализа):**
    - Персонализация есть: только у Travelata и Tours.Tutu (2 из 5)
    - Рекомендации есть: только у ViPN Travel и Tours.Tutu (2 из 5)
    - Истории просмотров нет ни у одного из 5 конкурентов 
- **Целевое значение:** Персонализация на основе:
    - Истории просмотров (ключевое конкурентное преимущество)
    - Поведенческих паттернов (клики, время на карточке)
    - Демографических данных
    - Сезонных предпочтений
- **Обоснование:** Отсутствие истории просмотров у всех конкурентов — это рыночная ниша

### Цель 5. Увеличение конверсии бронирования

- **Показатель:** Отношение завершённых бронирований к поисковым запросам
- **Текущее состояние:** Типичная конверсия агрегаторов туров — 1-3%
- **Целевое значение:** Увеличение конверсии на 20-30% за счёт персонализации
- **Обоснование:** Персонализированная выдача показывает релевантные туры, снижая когнитивную нагрузку на пользователя

### Цель 6. Улучшение качества работы фильтров

- **Показатель:** Время применения фильтров
- **Текущее состояние (по данным конкурентного анализа):**
    - Tours.Tutu: 2 сек 
    - OnlineTours: 11 сек
    - Travelata: 20 сек
    - Level.Travel: 25 сек
    - ViPN Travel: 23 сек
- **Целевое значение:** ≤ 3 секунды при любом количестве активных фильтров
- **Обоснование:** Индексация (Elasticsearch) и предвычисление результатов для популярных комбинаций фильтров

### Цель 7. Устранение проблем с пустыми результатами

- **Показатель:** Качество обработки запросов с нулевыми результатами
- **Текущее состояние (по данным конкурентного анализа):** Все 5 сервисов показывают статичное сообщение «не найдено» без альтернатив (за исключением либо отключения фильтров, либо выход на контакт с менеджером)
- **Целевое значение:** Система предлагает альтернативные варианты:
    - Расширение дат вылета (±3 дня)
    - Соседние курорты
    - Похожие отели по характеристикам
    - Изменение бюджета
- **Обоснование:** Умная система не оставляет пользователя без результата

# КОнцепция решения

«Интеллектуальная система персонализированного подбора и бронирования туров с адаптивной загрузкой контента на основе машинного обучения»

Конкурентный анализ 5 сервисов (Level.Travel, OnlineTours, Travelata, ViPN Travel, Tours.Tutu) выявил критические недостатки: ни один сервис не использует историю просмотров (0 из 5), персонализация реализована только у 2 из 5, наблюдается дисбаланс производительности (Level.Travel имеет быстрый API 96-205 мс, но загружает 22 MB за 55 сек; OnlineTours имеет лёгкую страницу 3-6 MB, но медленный API 904-1140 мс). Пользователи вынуждены вручную применять 15-20 фильтров и просматривать сотни нерелевантных вариантов, затрачивая 15-55 минут на подбор тура.

Решением этой проблемы является интеллектуальная бизнес-функция внедряет ML-алгоритмы (коллаборативная и контентная фильтрация) для персонализации выдачи на основе истории просмотров, поведенческих паттернов и предпочтений пользователя, а также реализует предиктивное кэширование и адаптивную загрузку контента. Это сократит время подбора тура с 15-55 минут до 5 минут, объём передаваемых данных с 22 MB до 3 MB, увеличит конверсию бронирования на 20-30% и обеспечит время ответа API ≤ 200 мс. Ключевое конкурентное преимущество — история просмотров, которой нет ни у одного из анализируемых сервисов.

# Ограничения

#### 1. Технические ограничения

- **Зависимость от внешних API туроператоров:** скорость и доступность данных определяются сторонними сервисами (Anex, Coral Travel, TUI и др.), SLA которых не контролируется нами
- **Объём данных:** база туров может содержать миллионы записей, что требует оптимизации хранения и индексации
- **Нагрузочные ограничения:** пиковые нагрузки в сезон (лето, новогодние праздники) могут превышать расчётную мощность системы

#### 2. Правовые ограничения

- **152-ФЗ «О персональных данных»:** сбор и хранение истории просмотров, поведенческих данных и демографии требует явного согласия пользователя
- **GDPR** (при работе с иностранными пользователями): дополнительные требования к хранению и обработке данных
- **Лицензионные ограничения:** использование отзывов и фотографий отелей возможно только при наличии договоров с правообладателями

#### 3. Ограничения по данным

- **Проблема холодного старта:** для новых пользователей без истории просмотров персонализация работает с низкой точностью
- **Качество данных:** неполные или устаревшие описания туров от операторов снижают качество рекомендаций
- **Сезонность:** модели, обученные на летних данных, могут некорректно работать зимой (и наоборот)

#### 4. Бизнес-ограничения

- **Бюджет на ML-инфраструктуру:** обучение и обслуживание моделей требует вычислительных ресурсов (GPU/TPU)
- **Сроки разработки:** полноценная ML-система требует итеративного обучения и A/B-тестирования, что увеличивает time-to-market
- **Интеграция с существующими системами:** необходимость совместимости с текущей CRM и биллингом компании

#### 5. Ограничения ML-моделей

- **Интерпретируемость:** сложные модели (нейросети) сложно объяснить пользователю («почему мне показали именно этот тур?»)
- **Дрейф данных:** предпочтения пользователей меняются со временем, модели требуют регулярного переобучения
- **Точность рекомендаций:** система должна предлагать релевантные туры не менее чем в 75% случаев. Для достижения этого показателя требуется достаточный объём данных о поведении пользователей (от 10 000 активных пользователей).

# Описание бизнес-процесса

## Модель AS-IS: текущий процесс подбора тура

Участники процесса: турист (конечный пользователь), веб-интерфейс агрегатора туров, серверная часть агрегатора, внешние API туроператоров, менеджер по продажам (опционально).

### Описание процесса

1. Авторизация на сайте агрегатора (Level.Travel, Travelata и аналогичных).
2. Ввод параметров поиска в ручном режиме: страна назначения, даты вылета и возвращения, количество туристов, бюджет.
3. Ожидание формирования результатов поиска — от 15 до 55 секунд согласно данным конкурентного анализа.
4. Просмотр первичной выдачи, содержащей от 100 до 914 предложений.
5. Ручное применение фильтров (от 15 до 20 наименований) со временем отклика от 2 до 25 секунд.
6. Итеративная корректировка параметров поиска при недостаточной релевантности выдачи.
7. Сравнительный анализ открытых карточек туров (в среднем 10–30 позиций).
8. Возврат к шагу 3 при невозможности выбора подходящего предложения.
9. Консультация с менеджером (при необходимости).
10. Оформление бронирования и оплата.

### Выявленные недостатки процесса AS-IS

- Продолжительность подбора тура составляет 15–55 минут.
- Высокая когнитивная нагрузка на пользователя вследствие необходимости анализа большого объёма нерелевантных данных.
- Отсутствие сохранения контекста поиска между сессиями: система не учитывает ранее просмотренные пользователем предложения.
- Избыточный расход сетевого трафика: от 3 до 26 МБ на одну сессию поиска.
- Отсутствие интеллектуальной обработки пустых результатов поиска.

## Модель TO-BE: целевой процесс с применением интеллектуальной бизнес-функции

Участники процесса: турист, интеллектуальная система подбора туров, ML-сервис персонализации, сервис кэширования, внешние API туроператоров.

### Описание процесса

1. Авторизация на сайте агрегатора.
2. Идентификация пользователя и загрузка профиля предпочтений:
    - при наличии истории взаимодействий — загрузка накопленного профиля;
    - для нового пользователя — использование геолокации, сезонных паттернов и статистики популярных направлений.
3. Ввод минимального набора параметров поиска (допускается запрос без указания конкретных критериев).
4. Формирование персонализированной выдачи за время не более 200 мс:
    - ML-модель выполняет ранжирование туров по степени соответствия профилю пользователя;
    - выполняется предзагрузка только топ-20 релевантных предложений (объём данных не более 3 МБ).
5. Отображение рекомендаций с пояснениями о причинах включения каждого предложения в выдачу.
6. Применение интеллектуальных фильтров со временем отклика не более 3 секунд; система автоматически предлагает только значимые для текущей выдачи фильтры.
7. При получении пустого результата система формирует набор альтернативных предложений: расширение диапазона дат (±3 дня), рассмотрение соседних курортов, подбор аналогичных отелей по характеристикам.
8. Обновление профиля предпочтений в режиме реального времени на основе действий пользователя.
9. Оформление бронирования с использованием предварительно заполненных данных.

### Преимущества процесса TO-BE

- Сокращение продолжительности подбора тура до 3–5 минут (в 5–10 раз по сравнению с AS-IS).
- Снижение когнитивной нагрузки на пользователя за счёт автоматизации фильтрации.
- Сокращение объёма передаваемых данных в 7 раз (с 22 МБ до 3 МБ).
- Формирование долгосрочного профиля предпочтений пользователя.
- Интеллектуальная обработка граничных случаев (пустые результаты поиска).

- Определены критерии приёмки для каждой пользовательской истории

- Глоссарий предметной области сформирован

- Определена архитектура системы и согласованы интерфейсы взаимодействия компонентов

**3. Данные для машинного обучения**

**1. Функциональная завершённость**
